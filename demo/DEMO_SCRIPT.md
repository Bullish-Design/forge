# Forge v2 Demo Script

Use this as the operator talk track while running:

```bash
devenv shell -- uv run demo/scripts/run_demo.py
```

## Step 1: Open The Live Site

Objective:
- Establish that the full stack is running and reachable on localhost.

Talking points:
- We are running the real `forge dev` orchestrator process, which starts:
  - `forge-overlay` as the public web entrypoint.
  - `obsidian-agent` for `/api/*` requests.
  - `kiln-fork` in watch mode with `--no-serve`.
- The site served to the browser is generated by kiln and wrapped by overlay.
- All updates are local and reproducible.

Operator actions:
- Open:
  - `http://127.0.0.1:18080/index.html`
  - `http://127.0.0.1:18080/references/kiln-capabilities`
  - `http://127.0.0.1:18080/experiments/live-reload`

## Step 2: Show Kiln Rendering Capabilities

Objective:
- Demonstrate rich markdown/static-site output from the sample vault.

Talking points:
- The vault includes tables, code fences, callouts, math, wikilinks, tags, folder pages, and canvas content.
- This confirms kiln rendering behavior on realistic notes, not synthetic minimal files.
- The capability matrix page maps each feature to a concrete artifact.

Operator actions:
- Walk through `index.html`, `canvas/roadmap`, `references/kiln-capabilities`, and `experiments/live-reload`.

## Step 3: Trigger Incremental Rebuild

Objective:
- Show watch-loop rebuilds from vault edits and webhook propagation into overlay.

Talking points:
- A token is appended to `experiments/live-reload.md`.
- `kiln-fork` detects the change and performs incremental rebuild.
- `--on-rebuild` posts webhook to overlay internal endpoint.
- Browser refresh shows newly rendered token.

Operator actions:
- Let the script mutate the note.
- Refresh `experiments/live-reload` when prompted.

## Step 4: Demonstrate Overlay API Proxy Health

Objective:
- Show `/api/*` requests are forwarded by overlay to backend.

Talking points:
- Browser/overlay clients call `/api/*` on overlay host.
- Overlay forwards to the real `obsidian-agent` service.
- This decouples UI from direct backend addressing.

Operator actions:
- Review printed JSON from `GET /api/health` via overlay.

## Step 5: Demonstrate Sync Bootstrap Endpoints

Objective:
- Show the new sync endpoint flow that Forge now uses at startup.

Talking points:
- Forge bootstrap sequence is ensure -> remote config -> sync -> status.
- In this demo, these are real obsidian-agent/obsidian-ops endpoints.
- This drives jj/git sync readiness, remote wiring, and initial fetch/rebase/push.

Operator actions:
- Review printed JSON for:
  - `POST /api/vault/vcs/sync/ensure`
  - `PUT /api/vault/vcs/sync/remote`
  - `POST /api/vault/vcs/sync`
  - `GET /api/vault/vcs/sync/status`

## Step 6: Demonstrate Apply Through Overlay

Objective:
- Show real agent mutation flow end-to-end.

Talking points:
- `POST /api/agent/apply` is sent to overlay.
- Overlay proxies to `obsidian-agent`.
- Agent updates `projects/forge-v2.md` via vault tools.
- Kiln rebuilds and rendered HTML updates reflect the mutation.

Operator actions:
- Review JSON apply response.
- Refresh `projects/forge-v2` and point out the apply token.

## Step 7: Demonstrate Undo Through Overlay

Objective:
- Show rollback behavior and resulting rebuild.

Talking points:
- `POST /api/undo` removes the last agent mutation.
- Rebuild happens automatically again.
- Rendered HTML no longer includes the apply token.

Operator actions:
- Review JSON undo response.
- Refresh `projects/forge-v2` and confirm marker removal.

## Step 8: Wrap-Up

Objective:
- Summarize proven behaviors and next-stage readiness.

Talking points:
- Confirmed kiln rendering + incremental rebuild behavior.
- Confirmed kiln-fork webhook integration (`--on-rebuild`) through overlay.
- Confirmed overlay injection and API proxy path.
- Confirmed sync bootstrap endpoint flow via overlay proxy.
- Confirmed apply/undo mutation lifecycle using real obsidian-agent wiring.

Operator actions:
